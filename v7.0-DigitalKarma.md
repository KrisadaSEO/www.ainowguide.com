## Purpose

Use this document as the practical implementation guide for any site in the Digital Karma portfolio that needs to reach v7.0 compliance.

## Canonical Definition of v7.0

Digital Karma Federation v7.0 is an open standard for AI-readable websites.

A v7.0 site is a site that:

- publishes a AI machine-readable `/ai/` layer
- exposes 6 required federation artifacts
- declares `federation_version` as `"7.0"`
- uses Schema.org structured data on primary pages
- publishes a Digital Karma score and federation relationship data
- provides an `llm.txt` file at the site root
- supports decentralized peer discovery through related federation sites

DigitalKarmaWeb.com is the specification authority and optional registry hub, but not a gatekeeper. Any site can independently implement the standard and operate as a federation member.

## What Changed in v7.0

Compared to v6.1, v7.0 formally changes the system in these ways:

- required federation artifacts increased from 4 to 6
- `/ai/federation.json` is now required
- `/llm.txt` is now required
- badge naming changes from `Karma Bronze` to `Karma Certified`
- compliance is now grouped into 4 formal tiers: Starter, Certified, Pro, Elite
- structured federation mapping in `/federation/` is introduced for advanced sites
- `/entities/` JSON-LD files are formalized for high-compliance sites
- `/ai/diagnostics.json` is introduced as optional but strongly recommended
- self-maintaining automation is documented for Elite implementations

## Minimum v7.0 Requirements

To declare a site as Digital Karma Federation v7.0, it must meet all of the following:

1. Expose all 6 required artifacts.
2. Keep all JSON valid and parseable.
3. Set `federation_version` to `"7.0"` anywhere that field appears.
4. Use HTTPS.
5. Include Schema.org structured data on all primary pages.
6. Publish a Digital Karma payload. For strict v7.0 compliance, this should include a score.
7. Link to at least one other federation member via `related_sites` or equivalent federation relationships.
8. Keep health and freshness data current, with weekly refresh as the baseline minimum.

## Required Artifacts

Every v7.0 site must publish these 6 files:

| Artifact | Path | Status | Purpose |
|---|---|---|---|
| Manifest | `/ai/manifest.json` | Required | Main identity and discovery entrypoint |
| Health | `/ai/health.json` | Required | Readiness and freshness state |
| Catalog | `/ai/catalog.json` | Required | AI machine-readable catalog of datasets and AI assets |
| Karma | `/ai/karma.json` | Required | Digital Karma definition or score payload |
| Federation | `/ai/federation.json` | Required | Federation role, peers, and topology |
| LLM Text | `/llm.txt` | Required | Plain-text LLM-readable site guide |

Notes:

- `compatibility.json` is currently retained in this repo for backward compatibility, but it is not one of the 6 required v7.0 artifacts.
- `llm.txt` lives at the site root, not in `/ai/`.

## Required File Definitions

### `/ai/manifest.json`

This is the primary discovery file. A crawler or agent should be able to start here and understand what the site is, what version it runs, and where the other endpoints live.

Required in practice:

- site name
- canonical site URL
- `federation_version`
- endpoint map or equivalent discoverable references
- updated timestamp

Recommended fields:

- tagline
- description
- `related_sites`
- datasets summary
- services
- contact or maintainer
- entities index reference
- agent prompt reference if using automation

Implementation note:

- The current repo's manifest is Schema.org-shaped and acts like a discovery catalog.
- Other sites do not need to copy the exact same prose, but they should preserve the function: identity, version, endpoint discovery, and federation context.

### `/ai/health.json`

This file reports operational readiness and content freshness.

Required in practice:

- canonical site identifier or homepage
- status
- updated timestamp
- basic metrics or checks

Recommended:

- last rebuild date
- endpoint presence checks
- counts for pages, datasets, entities, or cross-links
- explanatory notes

Status values should stay simple and AI machine-readable, for example:

- `ok`
- `ready`
- `degraded`
- `offline`
- `down`

Choose one internal convention per portfolio and use it consistently.

### `/ai/catalog.json`

This is the site's AI machine-readable catalog of datasets, endpoints, or other AI-facing resources.

Required in practice:

- `@context: "https://schema.org"`
- `@type: "DataCatalog"`
- name
- canonical URL
- `dataset` array

Each listed dataset or asset should include:

- `@type`
- name
- URL

Recommended:

- description
- `dateModified`
- `encodingFormat`
- dataset-specific metadata where available

### `/ai/karma.json`

This file defines the site's Digital Karma posture. In the strict v7.0 federation spec, this is a score-driven trust document. In the current repo implementation, it may also be used temporarily as a concept-definition file where dynamic scoring is not enabled.

Current portfolio rule:

- every site must publish `karma.json`
- if the site calculates a score, include `digital_karma_score`, signal breakdown, and badge
- if the site does not calculate live scoring yet, the file must still clearly define the site's Digital Karma state and version

Compliance note:

- a concept-only `karma.json` is acceptable as a transitional portfolio state
- a scored `karma.json` is the target for strict v7.0 compliance and is required for Certified, Pro, and Elite tier claims

Recommended score fields when scoring is enabled:

- site
- `federation_version`
- `digital_karma_score`
- signals
- badge or badges
- updated timestamp
- notes

### `/ai/federation.json`

This is the formal federation relationship document and is required in v7.0.

It should communicate:

- `federation_version`
- site role or node type
- network name
- canonical manifest authority
- site identity
- endpoint references
- related peers and relationship types
- status

Recommended relationship data:

- peer URL
- relationship type such as `peer`, `registry`, `parent`, `child`, `provider`, `consumer`
- trust level
- whether the relationship is bidirectional

### `/llm.txt`

This is the root-level AI-readable plain-text summary for the site.

It should include:

- site name
- short description
- canonical URL
- core pages
- datasets
- AI endpoints
- federation context
- current federation version

Best-practice characteristics:

- Markdown-style plain text
- concise and scannable
- relative paths or canonical absolute URLs
- no heavy marketing copy
- updated when key pages, endpoints, or positioning change

## Schema.org Requirement

All primary public-facing pages should include valid Schema.org markup, preferably JSON-LD.

Minimum expectation:

- homepage
- about page
- key service pages
- important content or knowledge-base pages
- any dataset landing pages

Most useful types across the portfolio:

- `Organization`
- `Brand`
- `WebSite`
- `WebPage`
- `Service`
- `Article`
- `Dataset`
- `DataCatalog`
- `BreadcrumbList`

Portfolio expectation:

- JSON endpoints should use Schema.org where appropriate
- page markup should describe real entities and content, not stuffed placeholder schema

## Digital Karma Score Definition

The v7.0 scoring model uses 7 public signals:

| Signal | Weight |
|---|---:|
| Schema Coverage | 20% |
| Content Freshness | 15% |
| AI Endpoints | 25% |
| Federation Presence | 15% |
| External Links | 10% |
| Technical Quality | 10% |
| Dataset Quality | 5% |

Score range:

- `0.00` to `1.00`

Badge thresholds:

- `Karma Certified` at `>= 0.70`
- `Karma Pro` at `>= 0.85`
- `Karma Elite` at `>= 0.95`

Important naming rule:

- do not use `Karma Bronze` on v7.0 sites
- rename it to `Karma Certified`

## Compliance Tiers

### Starter

Minimum bar:

- all 6 required artifacts exist
- JSON is valid
- `federation_version` is `7.0`

### Certified

Starter plus:

- Digital Karma score `>= 0.70`
- Schema.org markup on primary pages

### Pro

Certified plus:

- score `>= 0.85`
- published datasets
- bidirectional peer links

### Elite

Pro plus:

- score `>= 0.95`
- automation or self-maintaining workflow
- `/ai/diagnostics.json`
- `/entities/` files and index

## Advanced v7.0 Patterns

### `/entities/`

Use `/entities/` for explicit JSON-LD entity declarations.

Recommended files:

- `/entities/organization.jsonld`
- `/entities/brand.jsonld`
- `/entities/website.jsonld`
- `/entities/service.*.jsonld`

Use `/entities/index.json` to list them.

This is required for Elite compliance.

### `/federation/`

Advanced sites should publish structured federation mapping in:

- `/federation/v01_listings.json`
- `/federation/v02_clusters.json`
- `/federation/v03_relationships.json`
- `/federation/v04_propagation.json`
- `/federation/index.json`

This is recommended for Pro and required for Elite.

### Self-Maintaining Pattern

Elite sites should support repeatable automation to rebuild federation artifacts after content or structural changes.

Recommended automation outputs:

- refreshed timestamps
- refreshed catalog and dataset counts
- refreshed federation topology
- refreshed karma payload
- generated diagnostics
- daily or periodic journal entries

## Upgrade Checklist for Existing Sites

Use this exact sequence when moving a site to v7.0:

1. Confirm the site uses HTTPS and a stable canonical domain.
2. Add or validate `/ai/manifest.json`.
3. Add or validate `/ai/health.json`.
4. Add or validate `/ai/catalog.json`.
5. Add or validate `/ai/karma.json`.
6. Add or validate `/ai/federation.json`.
7. Add or refresh `/llm.txt`.
8. Change all version declarations from `6.1` or older to `7.0`.
9. Rename any `Karma Bronze` badge usage to `Karma Certified`.
10. Add `related_sites` or explicit federation peer relationships.
11. Add Schema.org JSON-LD to all primary pages.
12. Add at least one dataset if the site is targeting Pro or higher.
13. Add `/ai/diagnostics.json` and `/entities/` if the site is targeting Elite.
14. Validate JSON syntax, URLs, dates, and endpoint accessibility.

## Validation Checklist

Before marking a site as v7.0 complete, verify all of the following:

- `/ai/manifest.json` loads
- `/ai/health.json` loads
- `/ai/catalog.json` loads
- `/ai/karma.json` loads
- `/ai/federation.json` loads
- `/llm.txt` loads
- every JSON file parses cleanly
- canonical URLs are absolute where needed
- timestamps use ISO format
- `federation_version` is `7.0`
- Schema.org exists on primary pages
- federation peer links are present
- robots rules do not block intended AI discovery

## Portfolio Conventions

Across Digital Karma sites, keep these conventions consistent:

- prefer static files over dynamic systems when possible
- use absolute URLs for cross-site references
- keep naming stable across endpoints
- keep JSON clean, small, and easy to parse
- avoid turning `health.json` or `karma.json` into analytics dashboards
- use Digital Karma as a trust and machine-readability layer, not a vague marketing label

## Implementation Notes for This Repo

This repository currently reflects the active v7.0 direction, with a few practical nuances:

- `specs/federation-spec-v7.md` is the current spec authority
- older per-file specs still provide field-level guidance but have not all been renamed to v7.0 yet
- `/ai/compatibility.json` is still present for backward compatibility
- the live `karma.json` here is conceptual rather than fully dynamic-scored

That means sites should implement the v7.0 required surface area first, then decide whether their `karma.json` is:

- a calculated score document
- a transitional conceptual definition document

The long-term target for mature federation members is a scored, fully structured trust payload.
